# INTENT

The single source of truth for what this project is *for*. Specs in `specs/` must trace back to one of the goals here; if a proposed spec doesn't, either INTENT is incomplete (update it deliberately) or the spec is out of scope.

## Mission

Help a developer doing AI-assisted coding manage the storage cost of running many parallel workspaces — without losing the ability to spin up new ones, and without unreviewed destructive action.

## Problem

- AI coding tools (Claude Code, OpenCode, others) make it cheap to spawn workspaces. Each carries its own `node_modules/`, Python `site-packages/`, Docker layers, ephemeral build outputs, etc.
- On this machine, the historical `dev/` root sits inside Google Drive (`~/Library/CloudStorage/GoogleDrive-…/Documents/dev`) and is continuously syncing every dependency tree to the cloud. That is the **primary cleanup target**.
- A new `~/dev/` (local, *not* synced) is the future home. It currently holds only `clain-me` (this repo) and `cognee-me`. Everything else lives in the synced tree and needs triage.

## Goals (what success looks like)

1. **Categorical visibility.** Per workspace, the developer can see which subtrees fall into known classes (cache-managed, ephemeral, bytecode, workspace-source) and whether the workspace itself sits inside the synced tree. *Categorical* rather than *quantitative*: the class determines the right home and the right action, regardless of size.
2. **Deliberate execution.** Every destructive or relocating action is captured in an executable plan. Plans default to dry-run; actual execution is gated behind an explicit flag and currently behind a development-phase gate that a named future spec must lift. No silent `rm -rf`, no silent moves, no surprise rebuilds.
3. **Two action categories, fully covered.**
   - **delete + recreate** — for regenerable directories (venvs, `node_modules`, build outputs, bytecode). The plan captures the recreate command derived from the workspace's manifest (`pyproject.toml`, `package.json`, etc.).
   - **move + triage** — for workspace source code being relocated (e.g. from the synced tree to `~/dev/`). The plan captures the move command plus integrity smoke tests (hardcoded path embeddings, lock-file presence, broken symlinks, `.envrc` references).
4. **Deliberate consolidation.** Cache-managed directories belong in one canonical store *per ecosystem*, outside the synced tree (pnpm store, Pixi/uv caches, etc. — there is no universal "single cache"). The tool advises on placement and the environment variable / config command needed.
5. **Reuse by default going forward.** New workspaces use package managers that share storage (pnpm for JS/TS; Pixi or Rye/uv for Python) rather than reproducing full dependency trees per project.
6. **Portability.** A workspace can be moved between the synced tree and `~/dev/` (or archived) without losing the ability to rehydrate its environment.
7. **Honest sync hygiene.** The tool itself, and anything it generates, must not contribute to the Google Drive sync problem.
8. **Version-controlled, reviewable evolution.** This project tracks its own changes via git on a defined remote (GitHub) with a pull-request practice. Specs land via PR; the goal-advisor + tests are the review gates.

## Non-goals

- Building a general-purpose macOS cleaner. Mole and others already exist; we orchestrate, we don't replace.
- Hiding what is happening. Even when execution is enabled, every action is logged with the plan record that authorised it.
- Cloud-side deletion or Google Drive API manipulation. Local-side only.
- A universal cross-ecosystem cache. Ecosystems own their stores; we route correctly.

## Constraints to respect

- The user works across multiple AI coding CLIs/GUIs simultaneously. Nothing here should assume a single agent or shell.
- Anything generated by this project (caches, scratch, fixtures) must stay outside the Google Drive-synced tree.
- Plans and their execution must be auditable after the fact — the plan record plus a log line per action is the audit trail goal 2 requires.

## Out-of-scope until INTENT is updated

If a change being proposed is not in service of one of the goals above, stop and update INTENT first, or push back to the user. That is the whole point of having this file.
